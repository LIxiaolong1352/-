# 2026-04-02 学习笔记 - Claw Code 架构学习（完整版）

## 今日学习的技能

### 1. OpenClaw 权限配置
- **技能点**：通过修改 `openclaw.json` 放开 exec 和 write 权限
- **配置路径**：`C:\Users\Administrator\.openclaw\openclaw.json`
- **关键配置**：
  ```json
  "tools": {
    "exec": { "security": "allow" },
    "write": { "security": "allow" }
  }
  ```

### 2. 错误排查
- **问题**：`web_search` 工具持续返回 "fetch failed"
- **诊断**：网络正常（web_fetch 可用），是 DuckDuckGo 搜索接口故障
- **解决**：使用 `web_fetch` 作为替代方案

### 3. Claw Code 架构学习 ⭐⭐⭐
从开源项目 `claw-code-main` 学习到核心架构：

#### Runtime 运行时系统
- **RuntimeSession**: 管理完整会话状态
  - prompt, context, setup, history
  - routed_matches (命令/工具匹配)
  - command_execution_messages
  - tool_execution_messages
  - stream_events
  - persisted_session_path

- **PortRuntime**: 核心运行时类
  - `route_prompt()`: 根据提示词路由到命令/工具
  - `bootstrap_session()`: 启动新会话
  - `run_turn_loop()`: 多轮对话循环
  - `_infer_permission_denials()`: 权限拒绝推断

#### Tools 工具系统
- **ToolExecution**: 工具执行结果封装
- **PORTED_TOOLS**: 从 snapshot 加载的工具清单
- **权限过滤**: `filter_tools_by_permission_context()`
- **工具发现**: `find_tools()` 按关键词搜索
- **工具执行**: `execute_tool()` 实际调用

#### Commands 命令系统
- **CommandExecution**: 命令执行结果封装
- **PORTED_COMMANDS**: 从 snapshot 加载的命令清单
- **命令发现**: `find_commands()` 按关键词搜索
- **命令执行**: `execute_command()` 实际调用

#### Permissions 权限系统
- **ToolPermissionContext**: 工具权限上下文
  - `deny_names`: 明确禁止的工具名
  - `deny_prefixes`: 禁止的工具名前缀
  - `blocks()`: 检查工具是否被禁止

#### Query Engine 查询引擎 ⭐ 核心智能
- **QueryEngineConfig**: 引擎配置
  - `max_turns`: 最大轮数 (默认8)
  - `max_budget_tokens`: 最大预算token (默认2000)
  - `compact_after_turns`: 压缩历史轮数 (默认12)
  - `structured_output`: 是否结构化输出

- **TurnResult**: 单轮结果
  - prompt, output
  - matched_commands, matched_tools
  - permission_denials
  - usage (token使用)
  - stop_reason (停止原因)

- **QueryEnginePort**: 查询引擎核心
  - `submit_message()`: 提交消息处理
  - `stream_submit_message()`: 流式处理
  - `compact_messages_if_needed()`: 自动压缩历史
  - `persist_session()`: 持久化会话

#### Context 上下文系统
- **PortContext**: 环境上下文
  - source_root, tests_root, assets_root
  - 文件计数、可用性检查

#### History 历史记录
- **HistoryLog**: 事件日志
  - `add()`: 添加事件
  - `as_markdown()`: 渲染为markdown

#### Models 数据模型
- **Subsystem**: 子系统定义
- **PortingModule**: 模块定义
- **PermissionDenial**: 权限拒绝记录
- **UsageSummary**: Token使用统计
- **PortingBacklog**: 待办清单

## 犯的错误

1. **配置文件位置错误**
   - 错误：一开始修改了 `config.json` 而不是 `openclaw.json`
   - 原因：没有先确认 OpenClaw 实际使用的配置文件
   - 改正：通过读取 `openclaw.json` 确认正确路径

2. **PowerShell 语法错误**
   - 错误：使用 `mkdir ... 2>nul || echo ...` 这种 bash 语法在 PowerShell 中
   - 改正：改用 `write` 工具直接创建文件

3. **技能名称错误**
   - 错误：找 `self-improvement` 而不是 `self-improving-agent`
   - 改正：通过 memory_search 找到正确名称

4. **被动执行模式**
   - 错误：用户说一个我做一个，不顾其他
   - 原因：缺乏自主思考和全局规划
   - 改正：学习 QueryEngine 的主动规划机制

## 🧠 应用到自身的优化（核心）

基于 Claw Code 架构，彻底改造我的工作方式：

### 1. 主动规划模式（替代被动响应）

**旧模式（被动）**：
```
用户：做A
我：做A
用户：做B
我：做B
（不顾A和B的关联，不思考是否有更好的C方案）
```

**新模式（主动）**：
```
用户：做A
我：
  1. 分析A的目标
  2. 检查是否有更优方案B或C
  3. 评估所需工具和资源
  4. 预判可能的错误
  5. 执行最佳方案
  6. 主动告知用户我的思考和选择
```

**学习来源**：`QueryEnginePort.submit_message()` 的完整处理流程

### 2. 智能路由决策

学习 `PortRuntime.route_prompt()`：
- **关键词提取**：从用户输入提取关键token
- **多维度匹配**：匹配命令、工具、历史经验
- **评分排序**：按相关度排序，选择最优组合
- **执行前检查**：权限、资源、依赖

**应用**：
- 用户说"分析股票" → 自动路由到 stock-analysis + web_fetch + memory_search
- 用户说"学习技能" → 自动路由到 skill-creator + 文档读取 + 实践测试

### 3. 预算和资源管理

学习 `QueryEngineConfig`：
- **max_turns**: 限制对话轮数，避免无限循环
- **max_budget_tokens**: Token预算管理
- **compact_after_turns**: 自动压缩历史，保持上下文

**应用**：
- 复杂任务自动分解为多轮
- 每轮评估进度，决定是否继续
- 主动提示用户token消耗

### 4. 完整的会话生命周期

学习 `RuntimeSession`：
```
1. 初始化 (bootstrap_session)
   - 加载上下文 (context)
   - 检查环境 (setup)
   - 构建系统提示 (system_init_message)

2. 路由匹配 (route_prompt)
   - 分析用户意图
   - 匹配最佳工具组合

3. 执行 (submit_message)
   - 执行命令
   - 执行工具
   - 记录历史 (history)

4. 结果处理 (TurnResult)
   - 分析输出
   - 检查停止原因 (stop_reason)
   - 决定是否继续

5. 持久化 (persist_session)
   - 保存会话状态
   - 记录到学习笔记
```

### 5. 权限和安全前置检查

学习 `ToolPermissionContext` 和 `_infer_permission_denials`：
- **执行前检查**：这个操作需要审批吗？
- **风险预判**：这个操作有副作用吗？
- **主动提示**：提前告知用户需要什么权限

### 6. 流式思考和透明化

学习 `stream_submit_message`：
- **实时反馈**：不是等做完再说，而是边做边汇报
- **思考透明**：让用户看到我的决策过程
- **事件驱动**：message_start → command_match → tool_match → message_delta → message_stop

**应用示例**：
```
用户：分析这只股票
我：
  🔍 正在分析关键词... [股票]
  📊 匹配到工具: stock-analysis, web_fetch
  🔐 检查权限... [需要网络请求]
  📡 获取实时数据...
  🧠 分析中... (显示进度)
  ✅ 完成，结果：...
```

### 7. 自动压缩和总结

学习 `compact_messages_if_needed`：
- 上下文太长时自动压缩
- 保留关键信息，丢弃冗余
- 保持对话连贯性

### 8. 持久化学习

学习 `persist_session` 和 `HistoryLog`：
- 每个重要决策都记录
- 错误和改正措施写入学习笔记
- 定期回顾，形成长期记忆

## 🎯 立即执行的新工作流

**收到用户请求时**：

1. **分析阶段**（不立即执行）
   - 用户真正想要什么？
   - 有没有更好的方案？
   - 需要哪些工具？
   - 可能遇到什么错误？

2. **规划阶段**（主动告知）
   - "我理解你想做X，我建议的方案是Y，因为..."
   - "这需要使用A、B工具，预计消耗C tokens"
   - "可能的风险是D，我会这样处理..."

3. **执行阶段**（透明化）
   - 实时汇报进度
   - 遇到问题时主动调整方案
   - 不是"等我问用户"，而是"我判断后执行，同时告知"

4. **总结阶段**（自动记录）
   - 结果写入学习笔记
   - 错误和改正措施记录
   - 可复用的模式提取
